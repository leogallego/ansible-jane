# UI/UX Review - 2026-05-25

This report reviews the Jetpack Compose UI layer of Ansible Jane with a focus on:

- optimization (performance and maintainability),
- standardization (design-system consistency),
- modernization (current Material 3 / Compose best practices).

The review combines a manual pass over core screens and shared components plus parallel specialist audits (design system, accessibility/UX, Compose architecture).

## Scope

- `app/src/main/kotlin/io/github/leogallego/ansiblejane/ui/**`
- `app/src/main/kotlin/io/github/leogallego/ansiblejane/assistant/ui/**`
- `app/src/main/kotlin/io/github/leogallego/ansiblejane/ui/theme/**`
- `app/src/main/kotlin/io/github/leogallego/ansiblejane/navigation/**`
- `app/src/main/kotlin/io/github/leogallego/ansiblejane/MainActivity.kt`

## What Is Already Strong

- MVVM + `StateFlow` + `collectAsStateWithLifecycle` are used consistently in most screens.
- `LazyColumn` keys are generally present for paged data.
- Loading/error/empty states exist in all major feature areas.
- Reusable UI pieces already exist (`SearchBar`, `ErrorMessage`, `LoadingList`, `PaginationEffect`, `JobStatusBadge`).
- Material 3 baseline components are used throughout (`Scaffold`, `TopAppBar`, `NavigationBar`, `FilterChip`, `PullToRefreshBox`).

## Priority Findings

### High

1) **Pagination trigger can stall after first page**
- **Evidence:** `ui/components/PaginationEffect.kt` (`PaginationEffect`)
- **Issue:** `LaunchedEffect(shouldLoadMore)` only reruns when the boolean changes, so paging can stop when the user remains near the bottom and `shouldLoadMore` stays `true`.
- **Recommendation:** Key on `(shouldLoadMore, itemCount)` or use `snapshotFlow` with distinct filtering and a loading guard.

2) **Assistant message rendering does unnecessary heavy work on recomposition**
- **Evidence:** `assistant/ui/ChatBubble.kt` (`AssistantMessage`), `assistant/ui/AssistantScreen.kt` (`ActiveChatContent`)
- **Issue:** Markdown style objects and component wiring are rebuilt inline during message item composition, and new lambdas are created per item. Input changes can force expensive reprocessing in visible chat rows.
- **Recommendation:** Memoize markdown configuration with `remember(...)`, extract a stable `AssistantMarkdown` renderer, and hoist stable callbacks.

3) **Chat message actions are hidden behind long-press only**
- **Evidence:** `assistant/ui/ChatBubble.kt` (`UserBubble`, `AssistantMessage`, `MessageContextMenu`)
- **Issue:** Copy and regenerate actions are discoverable only via long press.
- **Recommendation:** Add visible per-message actions (inline or overflow button) and keep long-press as secondary.

4) **Edge-to-edge setup is inconsistent with content insets**
- **Evidence:** `MainActivity.kt` (`enableEdgeToEdge()`), `ui/main/MainScreen.kt` (`contentWindowInsets = WindowInsets(0, 0, 0, 0)`)
- **Issue:** System bars are enabled globally, but main scaffold insets are zeroed. This can cause overlap risks and inconsistent safe area handling.
- **Recommendation:** Use scaffold safe insets/defaults or apply `WindowInsets.safeDrawing` at the content root.

5) **Dashboard chart is not accessible to screen readers**
- **Evidence:** `ui/dashboard/DashboardScreen.kt` (`JobHistoryChart`)
- **Issue:** Data is drawn via `Canvas` without semantic summary, and section headers are not marked as headings.
- **Recommendation:** Add semantic descriptions for chart data and heading semantics for section titles.

### Medium

6) **Design tokens are not centralized (spacing/shapes)**
- **Evidence:** literals repeated in `ui/main/MainScreen.kt`, `ui/settings/GeneralTab.kt`, `ui/settings/SettingsScreen.kt`, `ui/components/ProviderSwitchChip.kt`, `ui/components/TemplateCard.kt`, others
- **Issue:** Repeated raw `dp` values and custom corner radii create visual drift and raise refactor cost.
- **Recommendation:** Add shared spacing and shape tokens (`Spacing`, app `Shapes`) and replace scattered literals.

7) **Typography scale is only partially defined**
- **Evidence:** `ui/theme/Type.kt`
- **Issue:** Only a subset of Material 3 text styles is customized while many screens use other styles (`titleSmall`, `labelLarge`, etc.), leading to mixed visual hierarchy.
- **Recommendation:** Define the full M3 typography scale (or explicitly document selective overrides and align usage).

8) **Status colors are duplicated and not fully theme-driven**
- **Evidence:** `ui/theme/StatusColors.kt`, `ui/dashboard/DashboardScreen.kt`, `ui/workflows/WorkflowNodeOrder.kt`, `ui/settings/AgentTab.kt`
- **Issue:** Multiple hardcoded status colors are used directly outside theme tokens.
- **Recommendation:** Route all status semantics through a single theme-aware source and remove duplicated hex colors from screens/components.

9) **Repeated paginated list scaffolding across many screens**
- **Evidence:** `ui/templates/TemplateListScreen.kt`, `ui/workflows/WorkflowTemplateListScreen.kt`, `ui/jobs/RecentJobsScreen.kt`, `ui/hosts/HostsScreen.kt`, `ui/inventory/InventoriesScreen.kt`, `ui/eda/EdaAuditScreen.kt`, `ui/schedules/SchedulesScreen.kt`
- **Issue:** Search + refresh + pagination + empty/loading/error composition is repeated with minor differences.
- **Recommendation:** Introduce a reusable paginated screen scaffold composable and keep screen-specific item content as slots.

10) **Detail screen scaffolds are duplicated**
- **Evidence:** `ui/jobs/JobStatusScreen.kt`, `ui/workflows/WorkflowJobStatusScreen.kt`, `ui/workflows/WorkflowTemplateDetailScreen.kt`, `ui/approval/ApprovalDetailScreen.kt`, `ui/settings/SettingsScreen.kt`
- **Issue:** Similar top app bar/back/padded content patterns are copied.
- **Recommendation:** Create a shared `DetailScaffold` for consistent navigation affordance, tags, and window inset behavior.

11) **Settings tab selector uses custom pills without native tab semantics**
- **Evidence:** `ui/settings/SettingsScreen.kt` (`SettingsTabSelector`)
- **Issue:** Visual tabs are custom clickable surfaces; accessibility selection/role semantics are weaker than native tab components.
- **Recommendation:** Use `TabRow`/`ScrollableTabRow` or add explicit tab semantics and selected state metadata.

12) **Provider switch chip likely below recommended touch target**
- **Evidence:** `ui/components/ProviderSwitchChip.kt`
- **Issue:** Compact custom chip layout (`padding(vertical = 6.dp)` + custom row) can fall under recommended minimum touch target.
- **Recommendation:** Move to `AssistChip` or enforce min touch target and clearer accessibility label semantics.

13) **Inconsistent empty/error recovery affordances**
- **Evidence:** `ui/components/EmptyState.kt`, `assistant/ui/AssistantScreen.kt`, `ui/notifications/NotificationsSheet.kt`
- **Issue:** Some screens provide actionable recovery (`ErrorMessage` retry), others are text-only dead ends.
- **Recommendation:** Standardize a shared status panel with title/body/action buttons for empty, error, and disconnected states.

14) **No `@Preview` coverage for critical UI components**
- **Evidence:** no `@Preview` usage in `app/src/main/kotlin`
- **Issue:** UI iteration and design regression detection are slower and rely only on runtime/testing.
- **Recommendation:** Add previews for shared components and key screen states (loading/error/empty/content).

15) **All UI strings are inline literals**
- **Evidence:** no `stringResource(...)` usage in `app/src/main/kotlin`
- **Issue:** Hardcoded strings reduce consistency, localization readiness, and maintainability.
- **Recommendation:** Migrate user-facing strings to resources, starting with shared components and navigation labels.

## Modernization Opportunities (Compose + Material 3)

- Replace custom chip/segmented patterns with standardized M3 navigation and chip variants where possible.
- Establish a UI foundation package:
  - `theme/Spacing.kt`
  - full `theme/Type.kt` scale
  - unified status tokens
  - shared card variants (`InfoCard`, `ListCard`, `SettingsCard`)
- Introduce reusable screen shells:
  - `PaginatedListScreen`
  - `DetailScaffold`
  - `StatusPanel` (loading/empty/error/disconnected)
- Add accessibility baseline:
  - heading semantics for section titles
  - live region semantics for assistant streaming state
  - meaningful dynamic `contentDescription` for badges and status updates
- Add preview catalog for key components and states.

## Suggested Implementation Plan

### Phase 1 (quick wins)
- Fix `PaginationEffect` trigger logic.
- Add visible copy/regenerate controls in chat.
- Apply scaffold safe insets in `MainScreen`.
- Add heading semantics in dashboard sections and chart summary semantics.

### Phase 2 (standardization)
- Introduce spacing + shape tokens.
- Complete typography mapping.
- Consolidate status colors and status pills.
- Extract `DetailScaffold` and `StatusPanel`.

### Phase 3 (modernization + maintainability)
- Build generic paginated screen scaffold and migrate repeated list screens.
- Refactor assistant markdown rendering with memoized config.
- Add `@Preview` coverage and migrate inline UI strings to resources.

## Expected Impact

- Lower UI bug surface and easier future feature work through shared scaffolds.
- More consistent Material 3 behavior and visual language across tabs/screens.
- Better assistant UX discoverability and accessibility compliance.
- Reduced recomposition cost in chat and list-heavy areas.
